## Summary

Two cross-cutting improvements to the configurator repo's plan authoring and review thoroughness:

1. **Requirement 1 — Atomic plan steps with rationale:** PLAN files generated by planning skills adopt a structured multi-line format. Each step is atomic (one reversible concern) and carries a mandatory rationale (**Why**), a checkable completion signal (**Done when**), and an explicit consumer list (**Consumers affected**). A top-level **Dependency & Consumer Map** is added so authors surface affected consumers at plan time.
2. **Requirement 2 — Mandatory dependency/consumer traversal:** The `architecture-review`, `code-review`, and `opentofu-explorer` subagents must **MANDATORILY** traverse dependency consumers before sign-off. Codebases use CodeGraph (`codegraph_callers` depth >= 2, `codegraph_impact`) with grep fallback; IaC uses `tofu graph` + grep (CodeGraph does NOT index HCL).

> **This PLAN itself models the target format** — every step below uses the atomic+rationale structure and the map above is the Dependency & Consumer Map.

### Step format (canonical)

```
- [ ] **N.M** <single atomic action — verb + target + outcome>
    — **Why:** <what this unblocks / why it must precede others>
    — **Done when:** <objective, checkable completion signal>
    — **Consumers affected:** <who depends on this; none if N/A>
```

## Dependency & Consumer Map

Surfaces which nodes each change depends on (predecessors) and who consumes each node (successors) so reviewers/execution know traversal order and blast radius. Execution flows top-to-bottom (Phase 1 → 2 → 3 → 4).

| Node (file under edit) | Depends on (must be done first) | Consumers (who depends on this node) | Blast radius if changed |
|------------------------|---------------------------------|--------------------------------------|-------------------------|
| `ticket-plan-workflow-skill` (format + map) | — (foundation) | `plan-execution-skill`, `plan-updater-skill`, `ticket-creation-subagent`, any future PLAN reviewer | Every PLAN generated hereafter |
| `plan-execution-skill` (parse atomic + rationale, surface consumers) | `ticket-plan-workflow-skill` format | `plan-updater-skill`, plan-execution runs | Execution behavior of all plans |
| `plan-updater-skill` (preserve lines, flag malformed) | `ticket-plan-workflow-skill` format | plan-updater runs, `ticket-creation-subagent` self-check | Progress tracking fidelity |
| `architecture-review-subagent` (mandatory traversal gate + plan atomicity) | — (independent of skill edits) | PLAN reviewers, `code-review-subagent` parity | Review verdict confidence |
| `code-review-subagent` (mandatory `codegraph_impact` + plan atomicity) | — (independent) | review pipeline, PR quality gates | Review verdict confidence |
| `opentofu-explorer-subagent` (IaC traversal section) | — (independent) | `opentofu-provisioning-workflow-skill`, IaC reviews | IaC change safety |
| `opentofu-provisioning-workflow-skill` (Dependency & Consumer Analysis subsection) | `opentofu-explorer-subagent` traversal approach | IaC provisioning runs | IaC change safety |
| `ticket-creation-subagent` (enforce + self-check blocks commit) | `ticket-plan-workflow-skill` + `plan-updater-skill` (malformed-flag reuse) | Every PLAN it generates | Quality gate for all new plans |

**Key ordering constraints:**
- Phase 1 must precede Phase 3 — `ticket-creation-subagent` enforces the format defined in `ticket-plan-workflow-skill` and reuses `plan-updater-skill`'s malformed-step flag.
- Phase 2 is independent of Phases 1 & 3 and may be interleaved. Within Phase 2: 2.1 and 2.2 are independent of each other; **2.3 must precede 2.4** (the provisioning skill mirrors the opentofu agent's traversal method).
- Phase 4 verification gates merge — it re-checks all 7 prior steps.
