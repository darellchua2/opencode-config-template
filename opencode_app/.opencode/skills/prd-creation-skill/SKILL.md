## What I do

I provide a structured Product Requirement Document (PRD) creation workflow:

1. **PRD Template** — 14 industry-standard core sections (Problem, Goals, Requirements, Metrics, etc.) with template placeholders and worked examples
2. **Optional Sections** — 3 add-on sections (UX/Design, Technical Architecture, Go-to-Market) with prompt-first inclusion logic
3. **docs/prd/ Naming Convention** — draft files (`PRD-draft-{slug}.md`) renamed to ticket-keyed files (`PRD-{key}.md`) after ticket creation
4. **PLAN Back-Linkage** — bidirectional traceability between PRD and PLAN files
5. **Discovery Interview Workflow** — prompt-first flow that gathers content section-by-section with user confirmation

## When to use me

Use this skill when:
- Planning a new feature that needs product-level scoping before implementation
- A user says "create prd", "product requirement", "feature spec", "write prd", "product doc"
- You need to separate product requirements (the "what" and "why") from implementation phases (the "how" and "when")
- You want a structured artifact that feeds into the PLAN file
- Reviewing or updating an existing PRD

**Trigger phrases**: "create prd", "product requirement", "feature spec", "write prd", "product doc", "PRD"

## PRD Template

Every PRD follows this structure. The header links to the PLAN file (filled when a ticket is created). Core sections 1-14 are always present. Optional sections (UX/Design, Technical Architecture, Go-to-Market) are included only when the user confirms during the discovery interview.

### Header

```markdown
# PRD: {Feature Name}

**Status**: Draft | In Review | Approved
**Author**: {name}
**Date**: {YYYY-MM-DD}
**PLAN**: PLANS/PLAN-{key}.md _(filled when ticket is created)_
```

### Core Sections (always present)

#### 1. Problem Statement

Describe the problem this feature solves and why it matters now.

```markdown
## Problem Statement

{What problem exists today? Who is affected? What is the cost of not solving it?}

**Example**: Users cannot reset their password if they lose access to their 
registered email. Support receives ~50 tickets/month for manual password resets, 
averaging 2 business days to resolve. This creates a poor user experience and 
an operational burden on the support team.
```

#### 2. Background / Context

Provide historical and technical context needed to understand the problem.

```markdown
## Background / Context

{What led to this problem? What existing systems or decisions are relevant? 
What prior attempts (if any) were made?}
```

#### 3. Goals & Non-Goals

Define what success looks like and explicitly what is out of scope.

```markdown
## Goals & Non-Goals

### Goals
- {Goal 1 — what this feature must achieve}
- {Goal 2}

### Non-Goals
- {Explicitly out of scope — prevents scope creep}
- {What this feature deliberately does NOT do}
```

#### 4. User Personas

Identify who benefits from this feature.

```markdown
## User Personas

### Persona 1: {Name} — {Role}
- **Context**: {When/where do they interact with this feature?}
- **Needs**: {What do they need to accomplish?}
- **Pain Points**: {What frustrates them today?}
```

#### 5. User Stories / Jobs-to-be-Done

Express requirements from the user's perspective.

```markdown
## User Stories / Jobs-to-be-Done

### User Stories
- As a {persona}, I want to {action}, so that I can {outcome}.
- As a {persona}, I want to {action}, so that I can {outcome}.

### Jobs-to-be-Done
- When {situation}, I want to {motivation}, so I can {expected outcome}.
```

#### 6. Functional Requirements

Detail what the system must do. This is the core of the PRD.

```markdown
## Functional Requirements

### FR-1: {Requirement Name}
- **Description**: {What the system must do}
- **Priority**: Must | Should | Could | Won't (MoSCoW)
- **Acceptance**: {How to verify this requirement is met}

**Example**: 
### FR-1: Password Reset via SMS
- **Description**: Users who have a verified phone number can request a 
  password reset code via SMS. The code expires after 10 minutes.
- **Priority**: Must
- **Acceptance**: User enters phone number → receives 6-digit code → 
  enters code → sets new password → redirected to login.
```

#### 7. Non-Functional Requirements

Specify quality attributes: performance, security, accessibility, etc.

```markdown
## Non-Functional Requirements

### NFR-1: Performance
- {Metric}: {Target} (e.g., "Password reset page loads < 2s on 3G")

### NFR-2: Security
- {Constraint}: {Detail} (e.g., "Reset codes are single-use and hashed at rest")

### NFR-3: Accessibility
- {Standard}: {Detail} (e.g., "WCAG 2.1 AA compliant")
```

#### 8. Success Metrics (KPIs)

Define measurable success criteria.

```markdown
## Success Metrics

| Metric | Baseline | Target | Measurement Method |
|--------|----------|--------|-------------------|
| {Metric name} | {Current value} | {Goal} | {How measured} |

**Example**:
| Support tickets for password reset | 50/month | < 10/month | Zendesk ticket reports |
```

#### 9. Scope — In / Out

Explicitly define what is included and excluded.

```markdown
## Scope

### In Scope
- {Deliverable 1}
- {Deliverable 2}

### Out of Scope
- {Explicitly excluded — may be future work}
```

#### 10. Acceptance Criteria

Define "done" at the PRD level (complements per-requirement acceptance).

```markdown
## Acceptance Criteria

- [ ] {Criterion 1 — high-level, testable}
- [ ] {Criterion 2}
- [ ] All functional requirements with priority "Must" are implemented
- [ ] All NFRs are validated
```

#### 11. Risks & Dependencies

Identify what could go wrong and what this depends on.

```markdown
## Risks & Dependencies

### Risks
| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| {Risk} | High/Med/Low | High/Med/Low | {How to mitigate} |

### Dependencies
- {Dependency 1 — team, service, or external factor}
```

#### 12. Open Questions

Track unresolved decisions that need stakeholder input.

```markdown
## Open Questions

- [ ] {Question 1} — **Owner**: {who decides} — **Needed by**: {phase/date}
- [ ] {Question 2}
```

#### 13. Timeline / Milestones

Outline expected phases and key dates.

```markdown
## Timeline / Milestones

| Milestone | Target Date | Dependencies |
|-----------|-------------|--------------|
| {Milestone 1} | {Date} | {Dependency} |
| Design review | {Date} | — |
| Implementation complete | {Date} | Design approved |
```

#### 14. References

Link to related documents, tickets, and research.

```markdown
## References

- {Link/title — related PRD, design doc, research, competitor analysis}
- {Issue/ticket link}
```

## Optional Sections

These sections are NOT included by default. During the discovery interview, the subagent prompts the user for each:

### UX / Design

**When to include**: User-facing features with visible UI changes.

**Prompt**: "Would you like to add the UX/Design section? It covers wireframes, user flows, and design system alignment."

```markdown
## UX / Design

### Wireframes / Mockups
- {Link to Figma/Sketch or inline description}

### User Flows
1. {Flow step 1} → {Flow step 2} → {Flow step 3}

### Design System Alignment
- {Components from design system to use}
- {New components needed}
```

### Technical Architecture

**When to include**: Features with non-trivial system design, data model changes, or new integrations.

**Prompt**: "Would you like to add the Technical Architecture section? It covers system diagrams, component interaction, data model, and API surface."

```markdown
## Technical Architecture

### System Diagram
{ASCII or link to architecture diagram}

### Component Interaction
- {Component A} → {Component B}: {protocol/format}

### Data Model
- {New/modified entities and fields}

### API Surface
- {New endpoints or contract changes}

### Make vs Buy
- {Build internally vs use third-party — rationale}
```

### Go-to-Market

**When to include**: Features that require rollout strategy, user communication, or have pricing implications.

**Prompt**: "Would you like to add the Go-to-Market section? It covers launch plan, feature flags, migration, and user communication."

```markdown
## Go-to-Market

### Rollout Strategy
- {Phased rollout, feature flag, or big-bang}

### Feature Flags
- {Flag name}: {purpose and rollout percentage}

### Migration Plan
- {How existing users/data transition}

### User Communication
- {Email, in-app notice, blog post}

### Success Monitoring
- {Dashboard/metrics to watch post-launch}
```

## docs/prd/ Naming Convention

### Draft (during planning, before ticket exists)

```
docs/prd/PRD-draft-{kebab-slug}.md
```

- Slug derived from PRD title (e.g., "User Authentication" → `user-authentication`)
- Created by `prd-specialist-subagent` during the discovery interview
- The `**PLAN**:` field in the header is `_(filled when ticket is created)_`

### Final (after ticket creation)

```
docs/prd/PRD-{ticket-key}.md
```

- Renamed via `git mv` by `ticket-creation-subagent` during Full workflow
- `git mv` preserves git history
- If the draft was never committed (just written to disk on the new branch), a plain `mv` + `git add` is used instead

### Bidirectional Linkage

| Direction | Field | Location |
|-----------|-------|----------|
| PRD → PLAN | `**PLAN**: PLANS/PLAN-{key}.md` | PRD header (filled at rename time) |
| PLAN → PRD | `**PRD**: docs/prd/PRD-{key}.md` | PLAN header (injected by ticket-creation-subagent) |

## Discovery Interview Workflow

The `prd-specialist-subagent` follows this prompt-first flow:

### Step 1: Gather Title & Overview
```
Subagent: "What is the title and brief overview of the feature you want a PRD for?"
User: {provides title and overview}
Subagent: "I'll use: Title: '{title}'. Proceed?"
```

### Step 2: Iterate Through Core Sections
```
For each of the 14 core sections:
  Subagent: "Next section: {Section Name}. {Purpose description}. What content should go here?"
  User: {provides content or says "skip" — skip is noted but section heading is kept}
  Subagent: "Got it. Here's what I'll write: {summary}. Correct?"
```

### Step 3: Prompt for Optional Sections
```
After all 14 core sections:
  Subagent: "Core sections complete. Would you like to add the UX/Design section? 
            It covers wireframes, user flows, and design system alignment."
  User: {yes/no}
  
  Subagent: "Would you like to add the Technical Architecture section?
            It covers system diagrams, data model, and API surface."
  User: {yes/no}
  
  Subagent: "Would you like to add the Go-to-Market section?
            It covers launch plan, feature flags, and user communication."
  User: {yes/no}
```

### Step 4: Confirm Full PRD Before Writing
```
Subagent: "Here's a summary of the PRD:
  - Title: {title}
  - Core sections: 14 (with content)
  - Optional sections: {list of included ones}
  - File: docs/prd/PRD-draft-{slug}.md
  
  Proceed to write?"
User: {yes → write file; no → revise}
```
