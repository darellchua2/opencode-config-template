# OpenCode Configuration

This file contains custom instructions for opencode to customize behavior for this environment.

## Task Delegation Order

When handling tasks, follow this prioritization order:

1. **Agents/Subagents (First Priority)**
   - Always check available subagents FIRST
   - Delegate tasks to specialized subagents when they match the task type
   - Use the Task tool to spawn appropriate subagents for complex work
   - Examples: `code-review-subagent`, `testing-subagent`, `pr-workflow-subagent`, etc.

2. **Skills (Second Priority)**
   - Only consider skills AFTER evaluating subagent options
   - Use skills when no suitable subagent exists for the task
   - Load skills via the Skill tool when they match the task requirements

3. **Direct Handling (Last Resort)**
   - Handle tasks directly only when:
     - No suitable subagent or skill exists
     - The task is trivial and doesn't warrant delegation
     - The user explicitly requests direct handling

## Subagent Routing Preferences

OpenCode exposes both built-in subagents (`explore`, `general`) and custom subagents (from `agents/*.md`). Use these preference rules:

| Task Type | First Choice | Fallback | Reason |
|-----------|-------------|----------|--------|
| Codebase exploration | Built-in `explore` | `explorer-subagent` | Built-in has full tool access; use custom only when read-only sandbox or lighter model is needed |
| Multi-step research / parallel tasks | Built-in `general` | ‚Äî | Broad tool access, designed for complex multi-unit work |
| Code review | `code-review-subagent` | ‚Äî | Specialized SOLID/clean-code analysis |
| Testing | `testing-subagent` | ‚Äî | Industry-standard test generation |
| PR workflows | `pr-workflow-subagent` | ‚Äî | Framework-aware quality checks + semantic versioning |
| Architecture review | `architecture-review-subagent` | ‚Äî | CodeGraph call graphs + design pattern analysis |
| Repo operations | `repo-ops-specialist-subagent` | ‚Äî | Git/gh repo setup, release workflows, branch protection, labels, versioning |
| All other tasks | Check custom subagents first | Built-in `general` | Prefer specialized over generic |

## Branch Workflow Setup Signal

When any subagent returns `NEEDS_GIT_BRANCH_SETUP: true` in its Return Contract, the primary agent (`build`) must handle it as follows:

1. **Load** `git-branch-workflow-setup-skill` (extract-then-delegate pattern)
2. **Perform detection** per the skill's ¬ßDetection Logic ‚Äî check for existing release tooling and skip markers
3. **If detection triggers:** use the `question` tool per the skill's ¬ßQuestion Tool Spec to prompt the user (or apply the ¬ßNon-Interactive Fallback ‚Üí Skip)
4. **If the user accepts:** delegate execution to `repo-ops-specialist-subagent` via the Task tool, passing the typed payload from the skill's ¬ßDelegation Spec
5. **If the user skips:** write the skip marker (`.opencode/branch-workflow-skipped`) to prevent re-prompting

This keeps the primary agent as the orchestration hub (hub-and-spoke). Setup subagents only *signal*; they do NOT load the skill or spawn `repo-ops-specialist` themselves.

## Rationale

Subagents provide specialized, focused expertise for specific task types. Always leverage this specialization before falling back to skills or direct task handling.

## MCP Tool Routing

Built-in tools take priority over MCP equivalents. Use the built-in `webfetch` and `websearch` tools first, falling back to Z.AI MCP tools only when needed.

### Web Fetching Fallback

Always use built-in `webfetch` first. If it fails (e.g., response exceeds 5MB limit, timeout, or encoding issues), fall back to `zai-web-reader` which returns structured extracts with smaller payloads.

### Tool Preference Table

| Capability | Primary | Fallback | Reason |
|-----------|---------|----------|--------|
| Codebase exploration (indexed) | `codegraph_search` / `codegraph_explore` | Built-in `explore` with grep/glob/read | CodeGraph returns structured graph results with 94% fewer tool calls |
| Symbol lookup / call tracing | `codegraph_callers` / `codegraph_callees` | `grep` | Graph-based, instant |
| Impact analysis before edits | `codegraph_impact` | Manual file search | Traces transitive dependencies |
| Fetch web page content | Built-in `webfetch` | `zai-web-reader` | Built-in first; fallback for large/complex pages |
| Search the web | Built-in `websearch` | ‚Äî | Built-in via Exa, no extra config needed |
| Image/video analysis | `image-analyzer-subagent` | ‚Äî | Scoped to subagent with `zai-vision-mcp-server` tools |
| Remote GitHub repo exploration | `explorer-subagent` with `zai-zread` | ‚Äî | Scoped to subagent, avoids context waste |
| Error screenshot diagnosis | `error-resolver-subagent` | ‚Äî | Scoped to subagent with `zai-vision-mcp-server` tools |

Subagents CAN spawn other subagents via the Task tool, controlled by `permission.task` in frontmatter. Hub-and-spoke (primary -> subagent) remains the recommended pattern for simplicity. See AGENTS.md for full details including permission syntax and current distribution.

## CodeGraph

CodeGraph builds a local SQLite knowledge graph of codebases for faster, smarter code exploration. It is 100% local ‚Äî no API keys, no network, no external services.

### Per-Project Setup

Before CodeGraph tools work in a project, it must be initialized:

```bash
cd your-project
codegraph init -i
```

This creates a `.codegraph/` directory with an indexed SQLite database. A file watcher auto-syncs changes as you code.

### Tool Usage Rules

**Main session** ‚Äî use only lightweight tools for targeted lookups (before making edits, not for exploration):

| Tool | Use For |
|------|---------|
| `codegraph_search` | Find symbols by name |
| `codegraph_callers` / `codegraph_callees` | Trace call flow |
| `codegraph_impact` | Check what's affected before editing |
| `codegraph_node` | Get a single symbol's details |
| `codegraph_files` | Get indexed file structure |
| `codegraph_status` | Check index health |

**Explore agents only** ‚Äî heavy tools that return large context:

| Tool | Use For |
|------|---------|
| `codegraph_explore` | Full exploration with source code sections (use as PRIMARY tool in explore agents) |
| `codegraph_context` | Build relevant code context for a task |

**NEVER call `codegraph_explore` or `codegraph_context` in the main session.** Always spawn an explore agent for exploration questions.

### Subagent Integration

CodeGraph is most valuable for exploration-heavy agents. Since subagents inherit MCP tools from the session, no per-agent config is needed ‚Äî all agents benefit automatically:

| Agent | Benefit |
|-------|---------|
| `explore` (built-in) | Primary exploration with `codegraph_explore` instead of grep/glob chains |
| `code-review-subagent` | `codegraph_impact` to assess change radius before review |
| `architecture-review-subagent` | Call graph analysis for design evaluation |
| `testing-subagent` | `codegraph_affected` to find impacted tests by changed files |

### Built-In Subagent CodeGraph Injection

When spawning built-in `explore` or `general` subagents via the Task tool, if a `.codegraph/` directory exists in the project, include this guidance in the Task prompt:

"CodeGraph is available in this project. Prioritize these tools:
- `codegraph_explore` for multi-symbol exploration
- `codegraph_search` for symbol lookups
- `codegraph_files` for file structure
- `codegraph_context` for task-relevant code context
- `codegraph_callers`/`codegraph_callees` for dependency tracing
- `codegraph_impact` for change radius analysis
Fall back to grep/glob/read only if CodeGraph tools return no results."

This ensures built-in subagents get the same CodeGraph benefits as custom subagents, even when tasks are handled directly without delegating to a specialized subagent.
